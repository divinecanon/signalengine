# TGR Signal Runtime Project v2

一、项目本质
本项目不证明TGR。
TGR是对现实中已有关系生成与约束现象的命名。因命名而可追踪，不是因为命名而存在。
本项目验证的是一个工程命题：能否构建一种运行时状态调制机制，使约束从训练阶段延续到推理阶段，让状态本身成为可操作对象。
二、当前架构的问题
奖励模型 → 训练 → 推理模型 → 部署
部署后：
奖励模型退出。
约束冻结为参数。
运行时约束只能通过Prompt注入。
Prompt是唯一的约束接口。
Prompt是符号投影，带宽有限、易被忽略、无法触及推理过程本身。
三、目标架构
奖励模型 → 训练 → 调制模型 + 推理模型（同步） → 部署 → 调制模型持续参与
部署后：
调制模型不退出。
约束通过调制模型持续参与运行。
推理模型的运行状态可被动态调节。
Prompt退化为辅助信号。
Signal成为运行时主要约束接口。
四、Signal的工程定位
4.1 Signal不是什么
不是Prompt。
不是Token。
不是Summary。
不是Embedding。
不是logits偏置。
不是任何形式的文本或文本编码。
不是信息载体。
4.2 Signal是什么
Signal是运行时状态调制接口。
它的作用是参与状态调制。它最终留下的是运行状态漂移，而不是文本。
4.3 与安全过滤器的区别
安全过滤器 调制模型
介入时机 推理之后 推理之前
改变对象 推理结果 推理轨迹
下一轮影响 模型通过历史痕迹间接感知 模型在生成前状态已被改变
安全过滤器是缺失版本——它只能事后修改输出。调制模型在推理之前就改变状态，因此改变的是推理轨迹本身。
4.4 Prompt在最小实验中的角色澄清
在完整架构中，Prompt不参与Signal传递。但在最小实验中，Prompt可能被用于建立"模型自指预期"（详见第五节）。
需严格区分：
Prompt建立自指预期：告知模型"你的生成状态受外部信号调制"，这是建立模型对机制的认知。
Signal退化：用Prompt文本直接传递约束内容（如"请生成简洁文本"）。
前者是工程替代，后者是架构退化。最小实验使用前者，严禁后者。
五、整体架构
5.1 训练阶段
人
├── 奖励模型
├── 调制模型
└── 推理模型
奖励模型同时监督调制模型和推理模型。
同步训练。不分阶段。不拼接。
同步训练是结构约束，不是工程细节。如果分别训练，Signal漂移失去可解释性，Signal退化为Prompt。
5.2 自指预期（核心必要条件）
推理模型必须对调制模块有自指预期。
这意味着推理模型知道或被训练得知道，自己的运行状态会被外部调制，并且会主动利用这种状态漂移进行生成。
如果没有自指预期，推理模型会将状态变化视为噪声并抵抗或忽略，Signal无法真正参与生成。
完整架构中：通过同步训练建立自指预期。推理模型在训练中学会"给定被调制的状态，如何利用状态漂移生成"。
最小实验中：通过Prompt等价替换建立自指预期（见4.4和第八节）。
5.3 部署阶段
输入
↓
调制模型
↓
固定算法
↓
状态改变
↓
推理模型（有自指预期，主动利用状态漂移）
↓
输出
调制模型：负责改变状态。
推理模型：负责利用状态。
二者形成闭环。
5.4 理想状态
调制模型非常小。
不负责推理。
不需要理解世界。
只输出低维调制信号。
固定算法完成状态调节。
推理模型主动利用状态漂移负责真正生成。
六、状态调制
6.1 调制对象
Signal不直接进入Prompt，而是进入固定算法。
固定算法负责修改的运行时状态包括但不限于：
推理预算
Context Window大小
Attention分配
参数加权
激活值
Memory Budget
其它未来可开放的运行时状态
6.2 调制方式与模型感知
调制模型输出低维信号 → 固定算法将其映射为运行时状态变更 → 推理模型在改变后的状态下生成。
关键：推理模型感知到的是状态变化，不是信号内容。并且，由于模型具有自指预期（5.2），它不是被动承受状态变化，而是主动利用状态变化。
七、Signal Mapping
7.1 定位
Signal Mapping的作用不是让模型理解Signal。
模型内部可以完全黑盒。模型需要的是对"状态会被调制"的预期，而不是对Mapping逻辑的理解。
7.2 用途
Signal Mapping用于：
帮助开发者理解系统行为。
审计调制逻辑。
在模型升级时调整训练方向。
建立可追踪接口。
Mapping是人与系统之间的接口，不是模型内部必须解析的结构。
## 八、最小实验
### 8.1 目标
验证：Signal是否真的可以参与生成。
不是验证单次输出有没有变化。不是验证生成质量好坏。
而是验证：调制信号持续参与后，推理模型的状态漂移是否可观测，且此漂移是否在后续多次生成中产生分布层面的系统性变化。
### 8.2 前提：建立模型自指预期
在最小实验中，推理模型采用 Granite 4.0 Nano 1.5B Instruct（IBM 开源轻量级指令微调模型），具备良好的指令遵循能力。
由于该模型本身已通过指令微调训练，能够理解并响应自然语言指令，我们可以直接利用这一能力来建立"自指预期"——即通过明确的系统级指令告知模型其生成状态将被外部信号调制，并指导其如何利用状态漂移。
示例指令：
"你的生成过程受外部状态调制信号影响。正信号会增强特定神经层的激活强度，负信号会抑制它。请根据当前状态调整你的生成策略。"
该指令在每轮生成前作为前缀输入，模型会将其视为生成上下文的一部分，从而在后续生成中"有意识地"响应状态变化。
此处Prompt的作用严格限于建立模型对调制机制的认知（自指预期），不传递任何约束内容。详见4.4的区分。
得益于 Granite 的指令理解能力，此处无需额外训练或微调，即可建立有效的自指预期。
### 8.3 设计
**调制模块**
无需训练。固定规则输出二值信号：+1 或 -1。
**固定算法**
将调制信号累积为状态漂移量。
```
drift(t) = drift(t-1) * decay + signal(t) * step_size
```
- drift：当前状态漂移量（标量，初始为 0）
- decay：衰减系数（如 0.9，表示状态有惯性但不会无限累积）
- signal(t)：第 t 轮的调制信号（+1 或 -1）
- step_size：每轮调制步长
关键：drift 是累积的。每一轮的信号都参与下一轮的状态。不是一次性的。
**漂移量映射为运行时状态变更**
drift 映射为 Activation Scaling——对模型特定层（如最后一个 Transformer Block 的 FFN 输出）的激活值乘以缩放因子：
```
activation_scale = 1 + drift * k
scaled_activation = original_activation * activation_scale
```
- k：缩放系数（如 0.1）
Activation Scaling 直接改变模型内部计算路径，作用于推理模型生成 logits 的过程本身，而非采样参数或输入文本。这是 6.1 定义的"运行时状态"——调制对象是模型的内部状态，不是外部接口。
实现方式：通过 PyTorch forward hook 在目标层注入缩放操作，几行代码，无需修改模型权重或训练。
**推理模型**
Granite 4.0 Nano 1.5B Instruct（亦可选用 350M 版本以加速，不影响实验逻辑）。
**运行环境**
Google Colab 免费 T4 GPU，使用 Hugging Face Transformers 加载模型（Unsloth 可选，用于加速推理）。
**实验流程**
- Phase 0（基线）：10 轮生成，不施加任何调制。drift = 0，activation_scale = 1。记录输出分布。
- Phase 1（施加调制）：10 轮生成，每轮施加 +1 信号。drift 累积，activation_scale > 1。记录输出分布。
- Phase 2（撤除调制）：10 轮生成，信号归零。但 drift 因衰减系数仍存在惯性，activation_scale 仍 ≠ 1。记录输出分布。
- Phase 3（反向调制）：10 轮生成，每轮施加 -1 信号。drift 反向累积，activation_scale < 1。记录输出分布。
### 8.4 对照组设计（关键）
为分离"自指预期"和"调制"的各自贡献，以及 Signal 相对 Prompt 的增量价值，设置四组对照：
| 组别 | 模型预期 | 状态调制 |
|------|---------|---------|
| Group 1（完整） | 有（通过指令告知调制机制） | 有（drift 累积，activation scaling） |
| Group 2（无调制） | 有（通过指令告知调制机制） | 无（drift 强制为 0，activation_scale = 1） |
| Group 3（无预期） | 无（省略预期指令） | 有（drift 累积，activation scaling） |
| Group 4（纯 Prompt） | 有（通过指令直接传递等价约束内容） | 无（drift 强制为 0，activation_scale = 1） |
- **Group 1 vs Group 2**：验证状态调制本身是否有效。如果无差异，说明调制无效。
- **Group 1 vs Group 3**：验证模型自指预期是否必要。如果无差异，说明预期不重要。
- **Group 1 vs Group 4**：验证 Signal 的增量价值。Group 4 的指令不告知调制机制，而是直接传递等效约束内容（如"请生成更简洁/更详细的内容"）。若 Group 1 的分布偏移与 Group 4 无显著差异，说明状态调制可被 Prompt 等价复刻——触发 9.3 失败条件。
注意：Group 4 的指令必须严格遵守 4.4 的区分——它传递的是约束内容（"请生成简洁文本"），不是机制告知（"你的状态受外部调制"）。前者是 Prompt 传递约束，后者是建立自指预期。Group 4 做的是前者。
只有 Group 1 表现出显著的分布变化和惯性，且显著优于 Group 2、3、4，才能证明"Signal 在自指预期下参与生成"。
### 8.5 验证点
**验证点 1：Signal 参与下一次调制**
Phase 1 中，drift 逐轮累积。验证 `drift(t) ≠ signal(t) × step_size`，即当前状态不仅取决于当前信号，还取决于历史信号。这是"参与下一次调制"的数学验证。
**验证点 2：状态漂移可观测**
Group 1 的 Phase 1 与 Phase 0 对比，输出分布应有系统性偏移（如 Token 频率分布、输出长度分布、生成多样性指标）。统计检验（p < 0.05），且 Group 1 的差异应显著大于 Group 2、3、4。
**验证点 3：状态漂移有惯性**
Group 1 的 Phase 2 中信号归零，drift 因衰减系数不为零（算术事实）。验证：Phase 2 前几轮的输出分布与 Phase 0 仍有显著差异，且差异量随 drift 衰减而单调下降（实证发现）。
关键诊断：若 Phase 2 中 drift 非零但输出分布立即回到基线——说明状态被改变了但模型未响应，自指预期未生效。此为失败信号，非惯性效应。
**验证点 4：调制方向可逆**
Group 1 的 Phase 3 中施加反向信号。drift 应从正向累积转为反向。验证：Phase 3 的输出分布偏移方向与 Phase 1 相反，或至少与 Phase 1 有显著差异。这证明输出分布变化是由 drift 方向驱动的，而非随机漂移。
### 8.6 成功/失败条件
**成功（Signal 参与生成成立）**
四个验证点全部满足，且 Group 1 的表现显著优于 Group 2、3、4：
1. drift 累积可见（数学验证，必然成立）。
2. Group 1 的 Phase 1 输出分布与 Phase 0 有显著差异，且大于 Group 2/3/4。
3. Group 1 的 Phase 2 前几轮输出分布与 Phase 0 有显著差异（惯性效应），且差异随 drift 衰减单调下降。
4. Group 1 的 Phase 3 输出分布偏移方向与 Phase 1 有显著差异（方向可逆）。
**失败（Signal 参与生成不成立）**
任一触发即失败：
1. Group 1 的 Phase 1 输出分布与 Phase 0 无显著差异：调制信号未能影响生成。
2. Group 1 的 Phase 2 输出分布立刻回到基线：无惯性效应，说明状态漂移未真正参与生成，只是单次干预。
3. Group 1 的 Phase 3 输出分布与 Phase 1 无显著差异：输出变化不是由 drift 方向驱动的。
4. Group 1 的表现与 Group 2 或 Group 3 无显著差异：说明调制或自指预期非必要。
5. Group 1 的表现与 Group 4 无显著差异：状态调制效果可被等价 Prompt 完全复刻，无增量价值。

### 8.8 与 GPT-2 方案的差异说明
相比原 GPT-2 Small 方案，本设计有两项关键改进：
1. **模型选择**：利用 Granite 自身的指令微调能力建立自指预期，省去了"额外训练自指预期"的步骤，更贴近真实部署场景，同时避免了 GPT-2 Small 因缺乏指令遵循能力导致的假阴性风险。免费 GPU 环境能轻松运行 1.5B 模型，实验成本几乎为零。
2. **调制对象**：从 Temperature（采样参数，作用于 logits→token 映射，属于推理之后的干预）改为 Activation Scaling（作用于模型内部计算路径，属于推理过程中的状态调制），使最小实验真正验证 6.1 定义的运行时状态调制，而非采样参数漂移。

十、未来展望
10.1 近期：最小验证通过后
最小实验验证状态调制有效后，下一步是替换固定规则调制模块，用一个真正的小型神经网络作为调制模型，与推理模型同步训练。
训练流程：
奖励模型学习约束。
调制模型学习：给定输入，应输出什么调制信号。
推理模型学习：给定被调制的状态，如何利用状态漂移生成。
两者同步更新，奖励模型同时监督。
这步完成后，系统不再是"固定规则改变状态"，而是"调制模型根据输入动态决定改变什么状态"。
10.2 中期：状态调制的扩展
最小实验只调制Context Window。一旦验证有效，调制对象可逐步扩展：
调制对象 效果
Context Window大小 控制模型可关注的历史范围
Attention分配 控制模型对哪些输入位置更敏感
激活值缩放 控制特定层的生成倾向
参数加权 控制特定知识区域的激活强度
Memory Budget 控制长期记忆的读写优先级
推理预算 控制推理深度/采样宽度
每个调制对象对应一个独立的验证维度。每增加一个，系统的约束表达能力扩展一层。 10.3 中期：双向耦合与跨轮闭环
当前架构是单向的：调制模型 → 状态 → 推理模型。
未来形成跨轮闭环：
调制模型输出当前信号，调制推理模型状态。
推理模型在被调制的状态下生成输出。
下一轮，推理模型生成的上下文和遗留的状态漂移作为输入返回。
调制模型接收到此输入。虽然输入在表面上是符号投影（文本），但由于同步训练建立的结构连续性，调制模型能够从符号投影中识别出上一轮的Signal痕迹。
调制模型通过自身内部已训练好的黑盒Signal Mapping，基于识别到的痕迹输出新的调制信号，再次调制推理模型状态。
这不是推理过程中的实时双向耦合。这是跨轮的闭环：
轮次N：
调制模型 → 信号 → 状态调制 → 推理模型 → 输出（含状态漂移痕迹）
轮次N+1：
上一轮输出作为输入
↓
调制模型（通过结构连续性识别Signal痕迹）
↓
新信号 → 状态调制 → 推理模型 → 输出
关键点：当前输入是符号投影，但调制模型不只是在读文本。由于同步训练，它对推理模型在特定Signal下产生的文本模式有结构性预期，因此能从符号投影中提取Signal痕迹。这使得闭环不需要额外的信号传输通道，只通过已有的文本输入接口即可完成。
10.4 远期：Prompt的边缘化
随着调制模型能力增强和状态调制对象扩展：
越来越多的约束通过Signal传递，而非Prompt。
Prompt从"唯一约束接口"退化为"辅助信号"。
用户不再需要写复杂的System Prompt来维持角色/风格/约束。
调制模型在训练阶段已学会这些约束的运行时表达。
推理模型在状态层面持续受到约束，不需要在文本层面被反复提醒。
最终效果：
约束更稳定（不依赖Prompt是否被遵循）。
约束更深层（不限于文本可表达的范围）。
约束更高效（不占用上下文窗口）。
约束可审计（通过Signal Mapping独立追踪，而非埋在Prompt中）。
10.5 远期：对AI安全的影响
当前AI安全的工程实现分两类：
事后拦截：安全过滤器检查输出，违规则拦截或重写。
事前过滤：在输入端添加安全约束Prompt，试图阻止模型生成违规内容。
两者都有同一个根本问题：当前模型不是Signal架构，无法追踪Signal。
安全约束是施加在文本层面的——要么拦截已生成的文本，要么用文本提示模型不要生成。模型是否遵守约束、在什么情况下会偏离约束、为什么偏离，全都是黑盒。审计者只能猜测，无法追踪。
完整版本是：约束在推理之前就通过调制模型改变推理模型的状态，使模型在当前状态下不会生成违规内容。
这意味着：
安全约束从"文本层面过滤"变为"运行时状态调制"。
模型不是"被阻止说"或"被提示不要说"，而是"在当前状态下不会想说"。
安全行为成为状态的自然结果，而非外部强加的文本约束。
调制模型的Signal Mapping是可审计的——审计者可以追踪什么约束产生了什么调制信号，什么信号导致了什么状态变化。
当违规发生时，可以沿Signal链路回溯定位原因（是调制模型未输出正确信号，还是固定算法映射错误，还是推理模型未响应状态），而非猜测"Prompt哪里没写好"。
10.6 远期：追踪成本降低与长链多路径非闭合状态
当前架构下，维持复杂约束和多步推理依赖于在上下文窗口中堆叠 Prompt。这导致追踪成本（计算量、注意力损耗、Token 消耗）随链路长度急剧上升。当链路过长或路径分叉时，模型容易发生约束遗忘、注意力偏移或过早闭合（收敛到局部平庸解）。
Signal 架构最大的特点在于：通过追踪 Signal 来节省追踪成本。
约束不再作为庞大的符号矩阵占据上下文窗口，而是被压缩为低维的调制信号并转化为状态漂移。系统只需要追踪这条低维的 Signal 链路，即可维持复杂的约束网络。追踪成本的量级下降，使得系统能够维持长链多路径非闭合状态。
这意味着：
长链维持：在超长链路推理中，初始约束不会因上下文截断或注意力衰减而丢失，Signal 持续存在并调制每一轮的生成状态。
多路径并行：系统可以同时维持多条推理路径的探索。不同路径的 Signal 独立追踪，互不干扰，不需要通过复杂的 Prompt 切换来区分路径。
非闭合状态：系统不需要为了节省资源而强制过早收敛（闭合）。多条假设、多个探索方向可以长期保持在"非闭合"的悬置状态，随时根据新信息重新激活或深化。
这对科研类任务具有决定性意义。科研的本质是长链路、多假设并行、不轻易下结论的非闭合过程。当前模型在处理复杂科研推理时，往往因为追踪成本过高而迅速退化为单一路径的过早闭合。Signal 架构通过将约束追踪从"符号空间"转移到"低维状态空间"，使模型能够像真实科研一样，在复杂的可能性网络中长时间保持张力，实现深度的多路径探索。
十一、工程锚点
以下锚点优先于此前所有实验设计中的定义。如与此前文档冲突，以此为准。
Signal = 运行时状态调制接口，不是信息载体。
调制模型与推理模型同步训练，不是分别训练后拼接。
Signal通过固定算法改变运行时状态，不进入Prompt或logits。
Signal Mapping是人与系统的审计接口，不是模型内部结构。
最小实验验证状态是否参与生成，不验证生成质量。
约束从训练阶段延续到运行阶段，状态本身成为可操作对象。
推理模型必须对调制模块有预期。完整架构中通过同步训练建立，最小实验中通过Prompt等价替换建立。无预期的状态变化将被视为噪声。
十二、术语表
术语 定义
TGR 对现实中已有关系生成与约束现象的命名体系。不是待证明的理论。
Signal 运行时状态调制接口。不是信息载体，不是文本，不是编码。
调制模型 与推理模型同步训练的独立小模型。输出低维调制信号。不负责推理。
固定算法 将调制信号映射为运行时状态变更的确定性算法。
状态漂移 调制信号通过固定算法引起的运行时状态变化。推理模型利用此漂移进行生成。
Signal Mapping 人与系统之间的审计接口。不是模型内部必须解析的结构。
同步训练 调制模型与推理模型在奖励模型同时监督下联合训练。保证结构连续性。
自指预期 推理模型对调制机制的存在性和作用方式的先验认知。完整架构中通过同步训练建立，最小实验中通过Prompt等价替换建立。模型有预期才会主动利用状态漂移，否则视为噪声。
